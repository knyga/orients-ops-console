# GitHub Dev Reporting

Answer engineering-activity questions using live GitHub data through this repo's CLI.

## Domain (must-know)

- Activity is read across the **active** repos of the org (`orients-ai`; archived repos and forks are excluded).
- **Commits** are counted on each repo's **default branch**, within `[start, end]` (UTC day boundaries — engineering activity has no field-timezone basis, unlike Vimeo/field-ops). Each commit carries additions/deletions; **net** = additions − deletions.
- **Pull requests** are bucketed independently: a PR counts as **opened** when `createdAt` is in-period and **merged** when `mergedAt` is in-period (the two windows are independent).
- Contributors are grouped by GitHub **login**; a commit with no linked account falls back to its author name and is flagged `unlinked`. **Bot** accounts are tallied but flagged, ranked after humans, and excluded from the human contributor count.

## When to use

Any question like: "who committed the most in May?", "how many PRs were merged last month?", "net lines added this sprint?", "which repos were most active?", "what did each person work on in GitHub?".

## How to use

Run the CLI (defaults to the current month, UTC, if you omit the dates):

```bash
npm run github -- --start 2026-05-01 --end 2026-05-31
```

It prints JSON (same shape as `GET /api/github` live mode wraps in `{ activity }`; the shaped summary is what `--write` persists):

- `contributors[]` — `{ key, login, displayName, isBot, unlinked, commits, additions, deletions, net, prsOpened, prsMerged }`, humans first then bots, by commits desc
- `repos[]` — per-repo `{ repo, commits, additions, deletions, net, prsOpened, prsMerged, activityScore }`, most active first
- `totals` — `{ repos, contributors, commits, additions, deletions, net, prsOpened, prsMerged }`

Answer per-person questions from `contributors`/`totals`; repo questions from `repos`. Add `--format table` for a human-readable view.

To persist a period as committed artifacts, add `--write` — it writes two sidecars under `reports/github/` keyed by period (`2026-05` for a single month, `start_end` otherwise): a lossless `<period>.json` (the web's render source — the shaped summary, plus `summaries?` when generated) and a flat `<period>.csv` (`contributor,commits,additions,deletions,net,prsOpened,prsMerged,summary`; the repo ranking stays in the JSON). The web renders the committed JSON via `GET /api/github?period=<key>` (period list at `?periods=1`); the current month can still be refreshed against live GitHub (`?refresh=1&start=&end=`).

Add `--summarize` (implies `--write`) to fill the `summary` column with a per-contributor occupation summary generated by Claude (claude-opus-4-8) from that person's PR titles + commit headlines. It needs `ANTHROPIC_API_KEY` set in `.env`.

### Monthly user-data requests → always activity + summary per contributor, via sonnet subagents

When the user asks for **monthly GitHub user data** (e.g. "what did everyone work on in May", "month by month from Jan"), the standing expectation is: **return both the per-contributor activity AND a per-contributor occupation summary, and write the artifacts** — for every month requested. Generate the summaries with **Claude Code sonnet subagents** (not the CLI's built-in `--summarize`, which uses opus), one subagent per month, then feed them back through the CLI so the table and committed report share them. The flow mirrors the Jira skill:

1. `npm run github -- --start <YYYY-MM-01> --end <YYYY-MM-DD> --dump-work` → per-contributor work items (`{accountId, displayName, tickets:[{key,summary}]}`, where `accountId` is the contributor's stable key `login:<login>`/`name:<author>`, `key` is `#<pr>`/`<short-oid>`, `summary` is the PR title / commit headline) as JSON. Save one file per month.
2. Dispatch one **sonnet** subagent per month (`Agent` tool, `model: sonnet`). Each reads its work file, writes a ≤200-word English occupation summary for each contributor, and writes a JSON object `{accountId: summary}` to a file. Use the occupation-summary guidance in `lib/occupationPrompt.ts` (group items into themes; keep Ukrainian proper nouns; prose only).
3. `npm run github -- --start … --end … --summaries-file <path> --format table` → the CLI loads those summaries (no Claude call), prints the table, and writes `reports/github/<period>.{json,csv}` with the `summary` column filled. `--summaries-file` implies `--write`.

`--summaries-file <path>` reads summaries from a JSON object (`contributor key → text`) instead of calling Claude. `--dump-work` prints the per-contributor work list as JSON and exits (no Claude, no CSV) — the subagents' input.

Dates are inclusive and must be `YYYY-MM-DD`. Missing `GH_ACCESS_TOKEN` makes the CLI exit non-zero with a clear message — tell the user to set it in `.env` (see `.env.example`).

## Out of scope

This reports commit/PR throughput and repo activity only.

⚠️ **All-zeros means token scope, not an empty period.** If every contributor shows zero commits and zero PRs across active repos, the `GH_ACCESS_TOKEN` almost certainly lacks `repo`/org-read scope (the org's repos are private). The CLI prints a stderr warning in this case. Do **not** report "no activity" — tell the user to widen the token scope. (See repo memory `github-activity-token-scope`.)
